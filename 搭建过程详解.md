# C++ 项目脚手架搭建过程详解

## 一、项目目标

这个脚手架旨在为 C++ 项目提供一条**标准化的 DevOps 开发流水线**，整合了四个核心环节：

| 环节       | 工具          | 解决的问题                                         |
| ---------- | ------------- | -------------------------------------------------- |
| 依赖管理   | vcpkg         | 自动下载、编译、管理第三方 C++ 库（如 fmt、gtest） |
| 构建系统   | CMake + Ninja | 跨平台构建，一次配置到处编译                       |
| 代码格式化 | Clang-Format  | 自动统一代码风格，减少团队协作摩擦                 |
| 静态分析   | Cppcheck      | 编译前发现潜在 bug、内存泄漏等                     |

最终效果：开发者克隆项目后，只需 **3 条命令** 就能完成依赖安装、构建和运行。

## 三、软件安装

### 3.1 CMake

```powershell
winget install Kitware.CMake --location "D:\Dev\CMake"
```

- **安装位置**：`D:\Dev\CMake\bin\cmake.exe`
- **版本**：4.3.4
- **说明**：CMake 是构建系统生成器，读取 `CMakeLists.txt` 生成实际的构建文件（Makefile 或 ninja.build）

#### CMake 软件 ≠ VS Code CMake Tools 插件

这是一个容易混淆的点。VS Code 里的 CMake Tools 插件只是一个**图形界面遥控器**，它底层还是调用 CMake 软件来干活。装了插件不等于装了 CMake 软件本身。

### 3.2 vcpkg

```powershell
git clone https://github.com/microsoft/vcpkg.git D:\Dev\vcpkg
cd D:\Dev\vcpkg
.\bootstrap-vcpkg.bat
```

- **安装位置**：`D:\Dev\vcpkg\vcpkg.exe`
- **版本**：2026-05-27
- **说明**：vcpkg 是微软的 C++ 包管理器。`bootstrap-vcpkg.bat` 会从 GitHub 下载预编译的 `vcpkg.exe`，比从源码编译更快

### 3.3 Cppcheck

```powershell
winget install Cppcheck.Cppcheck --location "D:\Dev\Cppcheck"
```

- **安装位置**：`D:\Dev\Cppcheck\cppcheck.exe`
- **版本**：2.21.0
- **说明**：Cppcheck 是 C/C++ 静态分析工具，能在编译前检测未使用变量、内存泄漏等问题

### 3.4 Ninja（额外安装）

原计划用 `MinGW Makefiles` 生成器，但因中文路径问题（详见 6.2 节），额外安装了 Ninja：

```powershell
winget install Ninja-build.Ninja --location "D:\Dev\Ninja"
```

- **安装位置**：`D:\Dev\Ninja\ninja.exe`
- **版本**：1.13.2

## 四、项目文件配置

最终项目结构：

```
test/
├── .clang-format           # Clang-Format 代码格式化规则
├── .gitignore               # Git 忽略规则
├── .vscode/
│   ├── extensions.json      # VS Code 推荐扩展列表
│   └── settings.json        # VS Code 工作区设置
├── CMakeLists.txt           # CMake 构建脚本
├── CMakePresets.json        # CMake 预设（编译器 + 环境变量）
├── vcpkg.json               # vcpkg 依赖清单
├── vcpkg-configuration.json # vcpkg 基线配置（版本锁定）
├── main.cpp                 # 主程序入口
└── tests/
    └── main_test.cpp        # Google Test 单元测试
```

### 4.1 vcpkg.json — 依赖清单

```json
{
    "dependencies": [
        "fmt",
        { "name": "gtest", "version>=": "1.17.0" }
    ]
}
```

**作用**：声明项目依赖。类比 Node.js 的 `package.json` 或 Python 的 `requirements.txt`。

**为什么 gtest 要限制版本**：旧版 gtest 在 MinGW 下编译可能失败，文档建议 `>= 1.17.0`。

### 4.2 vcpkg-configuration.json — 基线配置

由 `vcpkg new --application` 自动生成，包含一个 git commit hash 作为"基线"：

```json
{
    "default-registry": {
        "kind": "git",
        "baseline": "cae2989df743645fb2510971b7a45febd9a2c361",
        "repository": "https://github.com/microsoft/vcpkg"
    }
}
```

**作用**：锁定 vcpkg 官方仓库的一个特定版本，确保所有开发者安装的依赖版本一致。**这个文件必须由 vcpkg 命令生成，不能手写**，因为 baseline 是真实的 git commit hash。

### 4.3 CMakeLists.txt — 构建脚本

```cmake
cmake_minimum_required(VERSION 3.10)
list(APPEND CMAKE_PREFIX_PATH "${CMAKE_SOURCE_DIR}/vcpkg_installed/x64-mingw-dynamic")
project(HelloWorld)
find_package(fmt CONFIG REQUIRED)
find_package(GTest REQUIRED)
add_executable(HelloWorld main.cpp)
target_link_libraries(HelloWorld PRIVATE fmt::fmt)
add_executable(HelloWorldTests tests/main_test.cpp)
target_link_libraries(HelloWorldTests PRIVATE GTest::gtest_main fmt::fmt)
```

关键指令释义：

| 指令                                       | 含义                                       |
| ------------------------------------------ | ------------------------------------------ |
| `cmake_minimum_required(VERSION 3.10)`   | 要求 CMake ≥ 3.10，低于此版本直接报错     |
| `list(APPEND CMAKE_PREFIX_PATH ...)`     | 告诉 CMake 去 vcpkg 的安装目录搜索库文件   |
| `project(HelloWorld)`                    | 项目命名                                   |
| `find_package(fmt CONFIG REQUIRED)`      | 查找 fmt 库，`REQUIRED` 表示找不到就报错 |
| `add_executable(...)`                    | 声明可执行文件目标                         |
| `target_link_libraries(... PRIVATE ...)` | 链接库，`PRIVATE` 表示依赖不传播给下游   |

### 4.4 CMakePresets.json — 构建预设

```json
{
    "version": 8,
    "configurePresets": [
        {
            "name": "mingw64-gcc",
            "displayName": "GCC 8.1.0 x86_64-w64-mingw32",
            "binaryDir": "${sourceDir}/out/build/${presetName}",
            "generator": "Ninja",
            "cacheVariables": {
                "CMAKE_C_COMPILER": "D:/mingw64/bin/gcc.exe",
                "CMAKE_CXX_COMPILER": "D:/mingw64/bin/g++.exe",
                "CMAKE_BUILD_TYPE": "Debug"
            },
            "environment": {
                "VCPKG_ROOT": "D:/Dev/vcpkg/"
            }
        }
    ]
}
```

**与文档原版的关键差异**：

| 配置项      | 文档原版                          | 实际使用的                 | 原因                                  |
| ----------- | --------------------------------- | -------------------------- | ------------------------------------- |
| C 编译器    | `D:/Dev/TDM-GCC-64/bin/gcc.exe` | `D:/mingw64/bin/gcc.exe` | 实际环境用 MinGW-W64 而非 TDM-GCC     |
| C++ 编译器  | `D:/Dev/TDM-GCC-64/bin/g++.exe` | `D:/mingw64/bin/g++.exe` | 同上                                  |
| 生成器      | `MinGW Makefiles`               | `Ninja`                  | `mingw32-make` 无法处理含中文的路径 |
| preset 名称 | `TDM-GCC-64`                    | `mingw64-gcc`            | 反映实际编译器                        |

### 4.5 .clang-format — 代码格式化规则

核心规则说明：

| 规则                  | 值     | 效果                                                        |
| --------------------- | ------ | ----------------------------------------------------------- |
| `IndentWidth`       | 4      | 缩进 4 空格                                                 |
| `ColumnLimit`       | 80     | 每行最多 80 字符                                            |
| `BreakBeforeBraces` | Custom | 函数/struct/union 的大括号换行；if/for/while 的大括号不换行 |
| `PointerAlignment`  | Left   | 指针星号靠左：`int* p`                                    |
| `SortIncludes`      | true   | 自动排序`#include`                                        |

配合 VS Code 的 `"editor.formatOnSave": true`，每次保存时自动格式化。

### 4.6 .gitignore — 版本控制忽略

## 五、依赖安装过程

### 5.1 生成基线配置

```powershell
vcpkg new --application
```

这个命令生成了 `vcpkg-configuration.json`，其中包含一个真实的 vcpkg 仓库 commit baseline，用于锁定依赖版本。

> ⚠️ 注意：这个命令也会生成一个空的 `vcpkg.json`（会覆盖已有文件），所以操作时先备份原有的 `vcpkg.json`，生成基线后再把依赖内容写回去。

### 5.2 安装依赖

```powershell
vcpkg install --triplet x64-mingw-dynamic
```

vcpkg 的安装流程（以 fmt 为例）：

1. **搜索**：在已知的软件包列表中搜索 "fmt"
2. **下载**：从 GitHub 下载 fmt 源代码
3. **构建**：用本地编译器（MinGW GCC 8.1.0）编译 fmt
4. **安装**：将编译好的库文件和头文件放到 `vcpkg_installed/` 目录

最终安装的包：

| 包                 | 版本       | triplet                 |
| ------------------ | ---------- | ----------------------- |
| fmt                | 12.2.0     | x64-mingw-dynamic       |
| gtest              | 1.17.0     | x64-mingw-dynamic       |
| vcpkg-cmake        | 2024-04-23 | x64-windows（辅助工具） |
| vcpkg-cmake-config | 2024-05-23 | x64-windows（辅助工具） |

## 六、遇到的问题及解决方案

### 6.1 vcpkg 下载超时 / 403 Forbidden

**现象**：

```
Downloading 7z2602-x64.exe... Attempt 1 of 3, retrying download.
error: Download timed out.
```

后来改为：

```
error: curl operation failed with response code 403.
```

**根因**：

vcpkg 自动读取 Windows 系统的 IE 代理设置，你的代理配置是 `localhost:1080`。但 IE 代理设置中，`HTTPS_PROXY` 的值可能是 `https://localhost:1080`，而实际的代理软件（如 Clash、v2ray 等）是 **HTTP 代理**，不是 HTTPS 代理。

- `https://` 前缀告诉 vcpkg"这是一个 HTTPS 代理"，vcpkg 会用 CONNECT 隧道方式连接
- 但你的代理软件是 HTTP 代理，只接受 `http://` 前缀的请求
- 结果：HTTPS 连接失败 → GitHub 返回 403 或超时

**vcpkg 自己的提示也说得很清楚**：

> If you are using an HTTP proxy, but setting HTTPS_PROXY variable to `https://address:port`. This is not correct. Try setting `http://address:port` to both HTTP_PROXY and HTTPS_PROXY instead.

**解决方案**：

```powershell
$env:HTTP_PROXY = "http://localhost:1080"
$env:HTTPS_PROXY = "http://localhost:1080"
```

手动将两个代理变量都设为 `http://`（而非 `https://`），vcpkg 就能正常下载了。

### 6.2 mingw32-make 无法处理中文路径

**现象**：

```
mingw32-make.exe[2]: *** No rule to make target
'D:/Devops项目学习内容/Cmake构建项目内容/707472949/test/main.cpp',
needed by 'CMakeFiles/HelloWorld.dir/main.cpp.obj'.  Stop.
```

**根因**：

CMake 使用 `MinGW Makefiles` 生成器时，会生成 Makefile，其中包含文件的**绝对路径**。当前项目路径含有中文字符（`Devops项目学习内容`、`Cmake构建项目内容`），`mingw32-make` 无法正确解析含中文的路径。

**解决方案**：

改用 **Ninja** 构建系统。Ninja 对 Unicode 路径的支持比 MinGW Make 好得多。

```powershell
# 安装 Ninja
winget install Ninja-build.Ninja --location "D:\Dev\Ninja"

# CMakePresets.json 中修改
"generator": "Ninja"   // 原来是 "MinGW Makefiles"
```

### 6.3 Debug 版本链接了错误的 DLL

**现象**：

`HelloWorld.exe` 运行后没有输出（预期输出 "Hello World!"），但程序不报错、不崩溃。

**根因**：

用 `objdump -p HelloWorld.exe` 查看发现，Debug 构建链接的是 `libfmtd.dll`（带 `d` 后缀的 debug 版本），而之前只从 `vcpkg_installed/x64-mingw-dynamic/bin/` 拷贝了 release 版的 `libfmt.dll`（无 `d` 后缀）。

vcpkg 的 DLL 命名规则：

- Release 版：`libfmt.dll`
- Debug 版：`libfmtd.dll`（名称中多一个 `d`）

**解决方案**：

```powershell
# 同时拷贝 debug 和 release 目录下的所有 DLL
Copy-Item "vcpkg_installed/x64-mingw-dynamic/debug/bin/*.dll" "out/build/mingw64-gcc/"
Copy-Item "vcpkg_installed/x64-mingw-dynamic/bin/*.dll" "out/build/mingw64-gcc/"
```

## 七、构建与运行

### 构建命令

```powershell
# 第一步：配置（指定生成器、编译器、构建类型）
cmake -S . -B out/build/mingw64-gcc -G "Ninja" \
    -DCMAKE_C_COMPILER=D:/mingw64/bin/gcc.exe \
    -DCMAKE_CXX_COMPILER=D:/mingw64/bin/g++.exe \
    -DCMAKE_BUILD_TYPE=Debug

# 第二步：编译
cmake --build out/build/mingw64-gcc
```

或使用预设（一步到位）：

```powershell
cmake --preset mingw64-gcc
cmake --build --preset mingw64-gcc
```

## 十、关键经验总结

1. **`vcpkg-configuration.json` 不能手写**：baseline 是真实的 git commit hash，必须由 `vcpkg new --application` 生成
2. **代理变量用 `http://` 而非 `https://`**：大多数本地代理软件是 HTTP 代理，`https://` 前缀会让 vcpkg 走 CONNECT 隧道导致失败
3. **中文路径 + MinGW Make = 翻车**：`mingw32-make` 对 Unicode 路径支持差，用 Ninja 替代
4. **Debug 和 Release 的 DLL 不通用**：Debug 版 DLL 名多一个 `d`（如 `libfmtd.dll` vs `libfmt.dll`）
5. **CMake Tools 插件 ≠ CMake 软件**：插件只是 VS Code 的遥控器，CMake 软件本身必须单独安装
